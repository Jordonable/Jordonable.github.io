146.LRU缓存

Created2026-03-29|Updated2026-03-29
|Post Views:

💡 核心思想：哈希表 + 双向链表

LRU（最近最少使用）缓存的核心要求是：get 和 put 操作的时间复杂度必须都是 O(1)。

  • 为什么要用哈希表？ 满足 O(1) 的查找需求，直接通过 key 定位节点。
  • 为什么要用双向链表？ 满足 O(1) 的节点移动和删除需求。单链表删除节点需要找前驱节点（O(N)），而双向链表已知节点位置时，可以通过 pre 和 next 指针实现 O(1) 删除。

⚠️ 核心考点与易错点复盘

1. 为什么双向链表的节点 Node 既要存 val 也要存 key？

这是很多初学者都会忽略的盲区！ 直觉上，哈希表已经是 key -> Node 的映射了，Node 里似乎只需要存 val 就够了。 原因在于缓存淘汰逻辑（淘汰队尾节点时）： 当缓存容量达到上限，调用 put 插入新元素时，我们需要在双向链表中删除尾部节点（最久未使用的数据）。但仅仅在链表中删除是不够的，我们还必须在哈希表中把对应的映射也删掉。 如果没有在 Node 中存储 key，当从链表尾部拿到要淘汰的节点时，我们就无法知道这个节点在哈希表中对应的 key 是什么，也就无法执行 hash.remove(key)。

2. 核心辅助动作：makeNew (将节点标记为最新使用)

在 LRU 中，无论是 get（读取已有数据）还是 put（修改已有数据），都意味着该数据被再次访问了，需要提升它的优先级。 这就引出了一个非常核心的辅助方法：makeNew(key) 或者叫 moveToHead(node)。 操作拆解：

  1. 先把这个节点从当前位置抽离（cache.remove(x)）。

  2. 再把这个节点插到链表的头部（cache.addHead(x)）。

笔记体会：把这个动作单独封装成一个函数，会让主逻辑的 get 和 put 代码非常干净，避免代码重复。

3. get 和 put 操作的正确姿势（避免冗余开销）

在我最初的实现中，如果 put 遇到一个已经存在的 key，我采取了最暴力的做法： ❌ 错误做法： 先在链表里删掉旧节点 -> 重新 new 一个新节点放到头部 -> 把哈希表里的旧 key 删掉 -> 重新 put 新节点进哈希表。 问题所在： 这产生了大量不必要的对象创建（new Node）和哈希表操作，增加了 GC 压力。

✅ 优雅且正确的做法： 既然节点本身就在内存里，也挂在链表和哈希表上，直接原地修改 val，然后挪到头部即可。哈希表里的引用根本不需要变！

📝 标准代码骨架记忆法

写这道题时，按照以下顺序写最不容易乱：

  1. 定义内部类 Node：必须包含 pre, next, key, val。
  2. 定义内部类 DoubleList：
    • 必须包含伪头节点（head）和伪尾节点（tail），这样在增删时不用处理空指针的边界条件。
    • 实现核心方法：addHead, remove, rmTail。
  3. 实现 LRUCache 主类：
    • 初始化 HashMap、DoubleList 和 capacity。
    • 实现辅助方法 makeNew。
    • 最后实现 get（判空 -> makeNew -> 返回）和 put（分已存在和不存在两种情况）。
